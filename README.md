### 本サービスの概要（700文字以内）
FITNESS_TRACKERは、ジムでトレーニングする人のための筋トレ記録アプリです。既存の筋トレ記録アプリでは、重量・回数・セットといった数値は記録できても、マシンやラックの安全バーの調整位置といった「器具のセッティング」を記録できず、毎回トレーニング前に器具を調整し直す手間がありました。本アプリでは、器具設定も記録に含めることで、次回トレーニング時の準備の手間をなくし、安全かつ再現性のあるトレーニングを継続できることを目指します。想定ユーザーは、ジムでマシン・フリーウェイトを使ってトレーニングしている30代後半以上の社会人で、自分なりの記録スタイルを持っている人です。まずは自分自身、続けてジムで一緒にトレーニングしている仲間に使ってもらい、フィードバックを受けながら改善していく予定です。主な機能は、ユーザー登録・ログイン、種目の登録、ジム・マシンの登録と選択、トレーニング記録の作成(重量・回数・セット・メモ)、記録一覧・詳細の確認・編集です。本リリースでは、メールアドレス・パスワード変更、進捗の可視化、AIパーソナル機能の追加も予定しています。

## 1. サービス概要

FITNESS_TRACKERは、ジムでトレーニングする人のための筋トレ記録アプリ。
重量・回数・セットに加え、安全バーやラックの調整位置などの器具設定も一緒に記録できる。
ジム・マシンの仕様差にも対応し、毎回の準備の手間を減らしながら、安全かつ再現性のあるトレーニングの積み重ねを支える。

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

過去のトレーニング記録を見返して、次にやるセットの重量を決めようとした時に、既存の筋トレ記録アプリでは自分がやりたい記録方法（種目の分け方や記録の形式）に合わず、欲しい情報がすぐに見つからない／入力しづらいと感じた。
「自分はこう記録したい」というイメージがあるのに、アプリの仕様に合わせて記録の仕方を変えざるを得なかったことに違和感を覚えた。

### 2-2. なぜそれが「気になった」のか

重量を正しく更新していくには、毎回同じ動作・同じ条件でトレーニングを行う必要がある。安全バーやラックの位置がトレーニングごとに微妙にズレると、フォームや負荷のかかり方が変わってしまい、「重量が伸びた」のか「動作が変わっただけ」なのかが分からなくなる。
さらに、安全バーの位置が適切でないと、限界まで追い込んだときに器具が支えてくれず、ケガにつながるリスクもある。
つまり、自分にとって筋トレの記録は「数字を残すこと」だけでなく、**安全に・再現性を持ってトレーニングを継続するための土台**であり、そこが揺らぐことに強い違和感を感じた。

## 3. 課題の整理

### 3-1. 表に見えている困りごと

既存アプリの記録形式が自分のやりたい記録方法（種目・形式）と合わず、使いにくい。
特に、マシンやラックの安全バー・調整位置を記録する欄が無いため、毎回トレーニング前にその場で器具を調整し直す手間がかかる。

### 3-2. 本当に解決したい課題は何か

- なぜ困る？ → 自分の記録方法に合わないと、過去の記録を見返すときに必要な情報（重量・回数だけでなく安全バーやラックの位置）がすぐ見つからない
- なぜそれが困る？ → トレーニングを始める前に、毎回マシン・ラックの位置調整を一からやり直す手間が発生する
- なぜそれが困る？ → 準備に時間がかかり、トレーニングへの集中やテンポが途切れてしまう

→ 本当に解決したい課題：**「重量・回数だけでなく、安全バーやラックの位置などの器具設定も一緒に記録し、次回トレーニング時の準備の手間をなくすこと」**

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

- 年齢：30代後半以上
  - 理由：自分自身が「既存アプリの記録形式に合わない」と感じた当事者であり、このアプリは自分と同年代・同じ悩みを持つ層に最も使ってもらいやすい
- 生活・状況：日常的にジムで筋トレを行っており、種目ごとに自分なりの記録の付け方・分け方を持っている人
  - 理由：定型フォーマットではなく、自由度の高い記録ができることで、自分のやり方を崩さずに継続できる
- 環境：ジム通い（マシン・フリーウェイトを使ったトレーニング）
  - 理由：ジムでは種目数・マシンの種類が多く、種目ごとに記録したい項目（マシン名、重量設定の段階など）も変わるため、その柔軟性が必要

### 4-2. 自分とユーザーの距離

**自分自身**

→ 自分が既存アプリに感じた不満がそのまま開発の起点になっており、ユーザー像が最も具体的に想像できるため。

### 4-3. 実際に使われる可能性について

- まず最初に使ってくれそうな人は誰か？ → 自分自身。次に、ジムで一緒にトレーニングしている仲間
- その人は、どんな場面でこのアプリを思い出すか？ → トレーニング中、次のセットの重量を決めるとき／マシンやラックの位置を調整するとき
- 今の生活の中で、代わりに使っているものは何か？ → 既存の筋トレ記録アプリ（OVLDなど）、または合わずにメモアプリ等で代替記録
- それを置き換えてまで使う理由はあるか？ → 自分のやりたい記録形式にそのまま合わせられる点、特に器具設定まで記録・再現できる点

ジムで一緒にトレーニングしている仲間にも実際に使ってもらい、自分と同じような不満（記録形式の不一致、器具設定の手間）を感じているか、フィードバックをもらいながら改善していく想定。

### 4-4. 導入・継続の流れ

- きっかけ：「既存アプリが自分の記録方法に合わない」というモヤモヤを抱えたタイミング
- 最初の一歩：ユーザー登録 → 自分がよく行う種目を登録
- 継続理由：トレーニングのたびに記録すること自体が次回の重量設定に直結するため、使うこと自体に実利がある（習慣化しやすい）

## 5. 既存サービス・競合調査

### 5-1. 似たサービスの調査

- サービス名：OVLD（オヴルド）／最もシンプルな筋トレ管理アプリ
- URL：https://apps.apple.com/jp/app/最もシンプルな筋トレ管理-ovld-オヴルド/id1620809560
- どんなことができるか：セット数・重量・回数・部位などを片手で直感的に入力できることにこだわったトレーニング記録アプリで、週ごとの成長を種目別・部位別のグラフで自動集計できる。体重の何％をウェイトとして扱うかを設定する「体重負荷率」機能により自重トレーニングも記録できる。種目ごとにメモを残す機能もあるが、ジムやマシンには紐づいておらず、ユーザーレビューでも「前回のメモをコピーしたい」という要望が出ている。

### 5-2. それでも自分が作りたい理由

OVLDにも種目ごとのメモ機能はあるが、メモは種目単位での記録であり、ジムやマシンには紐づいていない。実際のユーザーレビューでも「前回のメモをコピーしたい」という要望が出ており、結局は過去の記録を自分で探してコピーする手間が残っている。
自分が欲しいのは「メモが書けること」自体ではなく、「ジム・マシン・種目を選んだ瞬間に、前回その組み合わせで使った器具設定が自動で出てくること」であり、そこにズレを感じたのが自作したい理由。

### 5-3. 差別化を「体験価値」で表現する

このアプリは、**「複数のジムやマシンを使い分けてトレーニングしている人」**が、**「マシンの前に立って、これから使う設定を確認する瞬間」**に一番助けるアプリ。

- 差別化ポイント1：マシンに「マシン設定メモ」を1つ持たせることで、探してコピーする必要がなく、マシンを選んだ瞬間に固定的な器具設定（座席位置・足の角度など）を確認できること。これはMVPの段階で実現する
- 差別化ポイント2：トレーニング記録がジム単位で管理されること。OVLDは種目を軸にした管理が中心で、ジムという単位で記録や器具設定をまとめて見ることはできない。複数のジムを使い分ける人にとって、「今日はこのジムでどう記録していたか」をジムごとに振り返れることは独立した価値になる
- （本リリースでの追加差別化）：種目×マシンの組み合わせに対して、過去のトレーニング記録のメモ（その日ごとの気づき）も自動で振り返れるようにする。マシン設定メモとは別に、過去のセッション内容まで遡れる体験を提供する

## 7. このアプリで実現すること

### MVPで作る機能

- ユーザー登録・ログイン（サインアップ／ログイン画面）
- 利用規約・プライバシーポリシーの提示（サインアップ画面内にリンク／モーダル表示し、同意を取得。専用画面は設けない）
- トレーニング日付の選択（ホーム画面に組み込み。選択した日付で記録作成フローを開始する。専用画面は設けない）
- ジムの選択（ホーム画面に組み込み。日付と同様、トレーニング1回につき1つのジムを選ぶ前提のため専用画面は設けない）
- 種目の登録（自分で自由に種目名を追加できる）（種目一覧／種目登録画面）。種目・ジム・マシンの編集機能はMVPでは作らず、追加・削除のみで対応する（編集機能を作ると画面が増えるため）
- ジムの登録（ホーム画面で未登録の場合に登録できる。ジム名のみ入力）
- マシンの登録（マシン選択画面から登録できる。ホームで選んだジムに対し、マシン名と「マシン設定メモ」を入力。ジム名は選択済みのため再入力不要）
- トレーニング記録時のマシン選択（ホームで選んだジム内のマシンを選ぶ。選択すると、そのマシンに登録された「マシン設定メモ」を読み取り専用で確認できる）
- トレーニング記録の作成（重量・回数・セット数・メモ）
  - メモは自由記述のテキスト形式。トレーニングごとに変わる内容（その日の調子、フォームの気づきなど）を記録する。マシン固有の固定的な設定（安全バー位置、座席位置、足の角度など）は、マシン設定メモ側で管理するため、ここでは重複して書く必要はない
- 過去の記録一覧表示・記録詳細の確認・編集（種目ごとの履歴を確認し、記録内容を編集できる）

### 本リリースで作る機能

- 種目×マシンの組み合わせに応じた、過去のトレーニングメモ（その日ごとの気づき）の振り返り機能（マシン設定メモとは別に、過去のセッション内容まで遡れるようにする）
- メールアドレス・パスワードの変更機能
- 進捗の可視化（グラフ表示など）
- AIパーソナル機能（記録データを基にした次回トレーニング内容・重量の提案など）

## 8. このアプリの懸念点とその対策

### 懸念点①

- 何が問題になりそうか：ジムやマシンごとに器具の仕様（ラックの段数・安全バーの規格など）が異なるため、あるジムで記録した位置情報が、別のジムやマシンではそのまま使えない
- なぜそう思うか：自分自身、複数のジムを利用する／ジムの設備が更新されることもあり、「位置」という情報がジムやマシンに強く依存することに気づいたため
- そのために考えている対策：記録をジム・マシン単位で紐づけて管理できるようにする（同じ「ベンチプレス」でも、利用するジム・マシンごとに別の設定として記録・参照できるようにする）

## 10. 技術スタック

### 10-1. 使用予定の技術

- フレームワーク：Ruby on Rails
- DB：PostgreSQL
- デプロイ先：AWS（EC2など、RUNTEQで学んだ構成）
- 使用予定ライブラリ：devise（ユーザー認証）

### 10-2. 技術選定の比較検討

| 候補 | 採用/不採用 | 理由 |
| --- | --- | --- |
| Ruby on Rails | 採用 | RUNTEQのカリキュラムで学習済みで、MVPを素早く形にできる |
| MySQL | 不採用 | RUNTEQの学習でRDS（MySQL2）の経験はあるが、今回は学習目的も兼ねてPostgreSQLを選択 |
| PostgreSQL | 採用 | RUNTEQの学習も兼ねて選択。MVPでは器具設定をテキスト型の自由記述メモとして持たせるが、将来的にマシンごとの設定項目を構造化したくなった場合にJSON型などへ拡張しやすい点も評価 |
| Heroku/Render | 不採用 | RUNTEQでAWS（EC2・RDS・VPCなど）のインフラ構成を学習済みのため、その経験を活かせるAWSを採用 |

- 今回チャレンジしたい点：deviseを使ったユーザー認証の実装。これまで概念は学んできたが、実際に組み込んで動かす経験を積みたい
- 不安な点：器具設定を自由記述メモにすることで実装はシンプルになるが、検索や集計（例: 同じ設定での記録だけ絞り込む）がしづらくなる点。本リリースでマシンごとの構造化が必要になった際の移行コストも考慮しておきたい

### 10-3. キャッチアップ不足の懸念

devise導入によるユーザー認証は新しいキャッチアップ要素のため、認証部分の実装にかかる時間が見積もりにくい懸念がある。MVPでは最低限の新規登録・ログイン・ログアウトに絞り、パスワードリセット等の付加機能は本リリース以降に回すことで、開発期間内に収める方針。
React/Next等のフロント技術は今回のMVPでは採用せず、本リリース後に組み込む方向で検討する。
